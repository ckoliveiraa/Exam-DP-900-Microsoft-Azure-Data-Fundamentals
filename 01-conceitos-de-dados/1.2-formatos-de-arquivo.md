# 1.2 — Formatos de Arquivo de Dados

> Quando os dados são guardados em **arquivos**, o formato escolhido afeta o **tamanho**, a **velocidade** de leitura/escrita, a **compressão** e a **compatibilidade**. Não existe formato "melhor" — existe o formato certo para cada cenário.

---

## Formatos Baseados em Texto

Legíveis por humanos, fáceis de gerar e amplamente compatíveis — mas ocupam mais espaço e são mais lentos de processar em grande escala.

### Texto delimitado (CSV / TSV)

- Cada linha é um registro; campos separados por um **delimitador** (vírgula no CSV, tabulação no TSV).
- A primeira linha costuma ser o **cabeçalho** com os nomes das colunas.
- **Universal** — abre em Excel, é lido por qualquer ferramenta.
- Não guarda tipos de dados nem hierarquia.

```csv
ClienteID,Nome,Cidade
1,Ana Souza,Recife
2,Bruno Lima,Natal
```

### JSON (*JavaScript Object Notation*)

- Estrutura **hierárquica** de pares chave-valor; suporta objetos e listas aninhados.
- **Autodescrito** e legível — padrão de APIs web e de bancos de documentos.
- Mais verboso que CSV.

```json
{ "clienteId": 1, "nome": "Ana Souza", "enderecos": [ { "cidade": "Recife" } ] }
```

### XML (*eXtensible Markup Language*)

- Hierárquico, baseado em **tags** de abertura e fechamento.
- Muito **verboso**; ainda comum em sistemas legados e troca de dados corporativa.

```xml
<cliente id="1"><nome>Ana Souza</nome><cidade>Recife</cidade></cliente>
```

---

## Formatos Otimizados (Big Data)

Formatos **binários**, projetados para processar grandes volumes com eficiência. Não são legíveis em um editor de texto, mas comprimem bem e são rápidos.

| Formato | Orientação | Compressão | Esquema | Melhor para |
|---------|-----------|------------|---------|-------------|
| **Avro** | **Linha** | Boa | Embutido (JSON) | **Escrita** intensiva, ingestão, streaming |
| **ORC** (*Optimized Row Columnar*) | **Coluna** | Muito boa | Embutido | Leitura analítica (ecossistema Hive) |
| **Parquet** | **Coluna** | Muito boa | Embutido | **Leitura analítica**, data lakes, Spark |

---

## Armazenamento em Linha × em Coluna

Essa é uma distinção que **cai no exame**.

| | Orientado a **linha** | Orientado a **coluna** |
|---|---|---|
| Como grava | Um registro inteiro por vez | Valores de uma mesma coluna juntos |
| Bom para | **Escrever** e ler registros completos | **Ler poucas colunas** de muitos registros |
| Compressão | Menor (tipos misturados) | Maior (valores do mesmo tipo juntos) |
| Carga de trabalho | Transacional (OLTP) | Analítica (OLAP) |
| Exemplo de formato | Avro | Parquet, ORC |

> **Por que coluna comprime melhor?** Valores de uma mesma coluna são do mesmo tipo e costumam se repetir — o que favorece muito os algoritmos de compressão.

---

## Como Escolher

| Cenário | Formato indicado |
|---------|------------------|
| Exportar/importar dados tabulares simples | **CSV / TSV** |
| Resposta de API ou documento NoSQL | **JSON** |
| Integração com sistema legado | **XML** |
| Ingerir/gravar muitos dados rapidamente | **Avro** |
| Consultar grandes volumes lendo poucas colunas | **Parquet / ORC** |

---

## Pontos-Chave para o Exame

- ✅ **CSV/TSV** = texto delimitado, simples, tabular, sem tipos nem hierarquia.
- ✅ **JSON/XML** = formatos de texto **hierárquicos** (semiestruturados).
- ✅ **Avro** = binário, orientado a **linha** → bom para **escrita/ingestão**.
- ✅ **Parquet e ORC** = binários, orientados a **coluna** → bons para **leitura analítica** e compressão.
- ✅ Coluna comprime melhor porque agrupa valores do **mesmo tipo**.
