# 1.1 — Como Representar Dados

> **Dados** são fatos sobre coisas, eventos e entidades do mundo real — nomes, quantidades, datas, imagens. Antes de armazená-los, é preciso entender **como eles são estruturados**, pois isso determina **onde**, **como** e **com qual tecnologia** serão guardados e consultados.

---

## Por que a representação importa

A forma como um dado é estruturado define:

- **O tipo de armazenamento** adequado (banco relacional, NoSQL ou arquivos).
- **Quando o esquema é aplicado** (na gravação ou na leitura).
- **Como o dado é consultado** (SQL, APIs, processamento de arquivos).
- **O custo e o desempenho** das operações.

---

## Os Três Tipos de Dados

| Tipo | Definição | Esquema | Exemplos |
|------|-----------|---------|----------|
| **Estruturado** | Segue um **esquema fixo** — tabelas com linhas, colunas e tipos definidos. | *Schema-on-write* | Tabela de clientes, planilhas, banco relacional |
| **Semiestruturado** | Tem estrutura, mas **flexível** — cada registro pode ter campos diferentes. | Esquema flexível / autodescrito | JSON, XML, documentos NoSQL |
| **Não estruturado** | **Sem esquema** — o conteúdo não se organiza em campos. | *Schema-on-read* (ou nenhum) | Imagens, vídeos, áudio, PDFs, e-mails |

---

## Dados Estruturados

Dados organizados em **tabelas** — linhas (registros/entidades) e colunas (atributos/campos).

- Cada coluna tem um **tipo de dado** definido (inteiro, texto, data...).
- O **esquema é definido antes** de inserir os dados — *schema-on-write*.
- Tabelas se ligam por **chaves**: primária (identifica o registro) e estrangeira (referencia outra tabela).
- Consultados com **SQL** (*Structured Query Language*).
- É o modelo dos **bancos relacionais**.

**Exemplo — tabela `Cliente`:**

| ClienteID | Nome | Cidade | DataCadastro |
|-----------|------|--------|--------------|
| 1 | Ana Souza | Recife | 2024-03-10 |
| 2 | Bruno Lima | Natal | 2024-05-22 |

> Todo registro tem **exatamente as mesmas colunas**. Inserir um cliente sem `Nome`, ou com um campo extra, viola o esquema.

---

## Dados Semiestruturados

Dados que **têm organização**, mas **não exigem esquema rígido**: cada registro carrega a própria estrutura (é "autodescrito").

- Usam **pares chave-valor** e **tags** para descrever o conteúdo.
- O conjunto de campos **pode variar** de um registro para outro.
- Suportam **hierarquia** (objetos dentro de objetos, listas).
- Formatos típicos: **JSON**, **XML**, **YAML**.
- Base dos bancos **NoSQL orientados a documento** (ex.: Azure Cosmos DB).

**Exemplo — dois clientes em JSON, com campos diferentes:**

```json
{ "id": 1, "nome": "Ana Souza", "telefones": ["81-9999", "81-8888"] }
{ "id": 2, "nome": "Bruno Lima", "email": "bruno@exemplo.com" }
```

> Ana tem dois telefones e nenhum e-mail; Bruno tem e-mail e nenhum telefone. **Ambos são válidos** — algo impossível numa tabela relacional sem colunas vazias.

---

## Dados Não Estruturados

Dados cujo conteúdo **não se encaixa em linhas e colunas**.

- Não há campos definidos — o significado está **dentro do próprio conteúdo**.
- O esquema, se houver, é aplicado **só na leitura/processamento** — *schema-on-read*.
- Geralmente armazenados como **arquivos ou blobs** (objetos binários).
- Exemplos: fotos, vídeos, áudios, documentos do Office, PDFs, e-mails, posts de redes sociais.
- No Azure, o destino típico é o **Azure Blob Storage** e os **data lakes**.

> Atenção: arquivos têm **metadados** (nome, tamanho, data), mas isso não torna o **conteúdo** estruturado.

---

## Schema-on-write × Schema-on-read

| | *Schema-on-write* | *Schema-on-read* |
|---|---|---|
| Quando o esquema é aplicado | Na **gravação** | Na **leitura** |
| Validação | Dado inválido é rejeitado ao inserir | Aceita qualquer dado; interpretado depois |
| Flexibilidade | Menor | Maior |
| Típico de | Bancos relacionais | Data lakes, arquivos não estruturados |

---

## Pontos-Chave para o Exame

- ✅ **Estruturado = esquema fixo + tabelas + SQL** (*schema-on-write*).
- ✅ **Semiestruturado = estrutura flexível e autodescrita** (JSON/XML) — campos variam por registro.
- ✅ **Não estruturado = sem esquema** (mídia, documentos) — guardado como arquivo/blob.
- ✅ Metadados de um arquivo **não** transformam seu conteúdo em estruturado.
- ✅ Saiba **classificar um exemplo** no tipo correto — formato comum de questão.
