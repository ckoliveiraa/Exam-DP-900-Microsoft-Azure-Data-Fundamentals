# 3.2 — Azure Blob Storage

> O **Azure Blob Storage** é o serviço do Azure para armazenar **dados não estruturados** (imagens, vídeos, documentos, backups, logs) como **objetos** chamados *blobs* (*Binary Large Objects*).

---

## Quando usar

Sempre que precisar guardar **arquivos** ou **dados não estruturados** em larga escala:

- Imagens e vídeos de sites e aplicativos
- Backups e cópias para recuperação de desastres
- Logs de aplicações
- Dados brutos para análise (base de **data lakes**)

> 💡 Se a pergunta menciona "armazenar imagens/vídeos/backups/arquivos" → a resposta é **Blob Storage**.

---

## Estrutura: Conta → Contêiner → Blob

| Nível | O que é |
|-------|---------|
| **Conta de armazenamento** | O nível superior — agrupa os serviços do Azure Storage. |
| **Contêiner** | Agrupa blobs (como uma "pasta" raiz). |
| **Blob** | O arquivo/objeto em si. |

![Hierarquia do Blob Storage: conta, contêineres e blobs](https://learn.microsoft.com/pt-br/azure/storage/blobs/media/storage-blobs-introduction/blob1.png)

---

## Camadas de Acesso (*Access Tiers*)

São o ponto **mais cobrado** dessa aula. Servem para **otimizar custo** conforme a frequência de acesso aos dados.

| Camada | Quando usar | Custo de armazenar | Custo de acessar |
|--------|-------------|--------------------|------------------|
| **Hot** | Dados acessados com **frequência** | Mais alto | Mais baixo |
| **Cool** | Acesso **raro** (mín. ~30 dias) | Menor | Maior |
| **Cold** | Acesso **muito raro** (mín. ~90 dias) | Ainda menor | Ainda maior |
| **Archive** | **Arquivamento** de longo prazo (mín. ~180 dias) | O mais baixo | O mais alto — exige **reidratação** |

> ⚠️ **Archive é offline.** Os dados não podem ser lidos diretamente — precisam ser "reidratados" (movidos para Hot ou Cool), o que leva horas. Se o cenário exige **acesso imediato**, **não use Archive**.

### Regra prática

- Acessado **toda hora** → **Hot**
- Acessado **uma vez por mês** → **Cool**
- Acessado **algumas vezes por ano** → **Cold**
- **Quase nunca** acessado, só por compliance/histórico → **Archive**

---

## Gerenciamento de Ciclo de Vida

Permite criar **regras automáticas** para mover blobs entre camadas ou excluí-los após certo tempo.

**Exemplo**: "mover blobs da camada Hot para Cool após 30 dias e para Archive após 180 dias" — tudo sem intervenção manual, reduzindo custos.

---

## Azure Data Lake Storage Gen2

É o **Blob Storage com um recurso extra**: o **namespace hierárquico** (pastas de verdade, não simuladas).

- Voltado para **análise de big data** (Spark, Databricks, Synapse).
- Mantém todos os recursos do Blob (camadas, redundância, ciclo de vida).

> 💡 Se o cenário fala em **big data analytics com estrutura de pastas/diretórios**, a resposta é **Data Lake Storage Gen2** — não Blob puro.

---

## Pontos-Chave para o Exame

- ✅ Blob Storage = **dados não estruturados** como objetos (imagens, vídeos, backups, logs).
- ✅ Hierarquia: **conta → contêiner → blob**.
- ✅ Camadas: **Hot / Cool / Cold / Archive** — escolher pelo equilíbrio **custo × frequência de acesso**.
- ✅ **Archive** é a mais barata, mas **exige reidratação** e não serve para acesso imediato.
- ✅ **Ciclo de vida** automatiza a movimentação entre camadas.
- ✅ **Data Lake Storage Gen2** = Blob + namespace hierárquico, para **analytics de big data**.
