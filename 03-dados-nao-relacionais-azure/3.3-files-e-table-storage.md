# 3.3 — Azure Files e Table Storage

> Além do Blob, a conta de armazenamento oferece o **Azure Files** (compartilhamento de arquivos pela rede) e o **Azure Table Storage** (banco NoSQL de chave-valor).

---

## Azure Files

Cria **compartilhamentos de arquivos** na nuvem, acessíveis por vários computadores e aplicações ao mesmo tempo, como uma "pasta de rede".

**Características:**

- Acessível pelos protocolos **SMB** e **NFS** (padrões de compartilhamento de arquivos).
- Pode ser **montado** como uma unidade de rede em Windows, Linux e macOS.
- Totalmente gerenciado — sem servidor de arquivos para manter.
- Acessível tanto da nuvem quanto de máquinas locais.

**Casos de uso:**

- Substituir servidores de arquivos locais.
- Compartilhar arquivos de configuração entre várias VMs.
- Armazenar ferramentas e logs acessados por várias máquinas.

---

## Azure Table Storage

Um banco de dados **NoSQL de chave-valor** para guardar grandes volumes de dados **estruturados, porém sem esquema fixo**.

**Características:**

- Cada item é uma **entidade** com um conjunto de propriedades (pares chave-valor).
- **Não exige esquema** — entidades da mesma tabela podem ter propriedades diferentes.
- Não há junções, chaves estrangeiras, stored procedures nem relacionamentos.
- Rápido e de baixo custo para grandes volumes.

### Chave da Entidade: Partition Key + Row Key

Cada entidade é identificada por **duas chaves**:

| Chave | Função |
|-------|--------|
| **Partition Key** (chave de partição) | Agrupa entidades relacionadas na mesma partição → escala e desempenho. |
| **Row Key** (chave de linha) | Identifica a entidade **dentro** da partição. |

> A combinação **Partition Key + Row Key** identifica unicamente cada entidade e é a forma mais rápida de consultá-la.

### Exemplo prático

Diferente de um banco SQL — onde todas as linhas têm as **mesmas colunas** — no Table Storage cada entidade é um **conjunto independente de pares chave-valor** e pode ter propriedades **diferentes** das outras.

Veja uma tabela de **pedidos** particionada por cliente:

```
┌─ Entidade 1 ──────────────────────────────┐
│ PartitionKey : "cliente-001"              │
│ RowKey       : "pedido-1001"              │
│ Produto      : "Notebook"                 │
│ Valor        : 4500                       │
│ Status       : "Entregue"                 │
└───────────────────────────────────────────┘

┌─ Entidade 2 ──────────────────────────────┐
│ PartitionKey : "cliente-001"              │
│ RowKey       : "pedido-1002"              │
│ Produto      : "Mouse"                    │
│ Valor        : 80                         │
│ Status       : "Enviado"                  │
│ Desconto     : 10        ← campo extra    │
└───────────────────────────────────────────┘

┌─ Entidade 3 ──────────────────────────────┐
│ PartitionKey : "cliente-002"              │
│ RowKey       : "pedido-2001"              │
│ Produto      : "Teclado"                  │
│ Valor        : 250                        │
│ Observacao   : "Presente"  ← campo extra  │
└───────────────────────────────────────────┘
```

- Todos os pedidos do **cliente-001** ficam na **mesma partição** → consulta rápida de "pedidos desse cliente".
- A **RowKey** (`pedido-1001`) identifica o pedido **dentro** daquela partição.
- A dupla `PartitionKey + RowKey` é **única** em toda a tabela.
- Cada entidade tem **propriedades diferentes** (uma tem `Desconto`, outra tem `Observacao`, outra não tem `Status`) — isso é o **schemaless** do NoSQL.

**Casos de uso:**

- Dados de catálogos de produtos, perfis de usuários, dados de IoT.
- Cenários que exigem leitura rápida por chave e custo baixo.

---

## Azure Files × Table Storage

| | Azure Files | Table Storage |
|---|---|---|
| Tipo de dado | Arquivos | Dados de chave-valor (NoSQL) |
| Acesso | Protocolos SMB / NFS | API / consulta por chave |
| Equivalente a | Pasta de rede compartilhada | Banco NoSQL simples |
| Esquema | Não se aplica | Sem esquema fixo |

---

## Pontos-Chave para o Exame

- ✅ **Azure Files** = compartilhamento de arquivos via **SMB/NFS**, montável como unidade de rede.
- ✅ **Table Storage** = NoSQL de **chave-valor**, sem esquema, sem junções.
- ✅ No Table Storage, a entidade é identificada por **Partition Key + Row Key**.
- ✅ A **Partition Key** agrupa dados e influencia desempenho e escala.
